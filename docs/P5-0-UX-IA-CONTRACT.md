# P5-0 Lead Pipeline UX/IA Contract

> **문서 상태**: Draft v1.0
> **작성일**: 2026-03-21
> **범위**: Outbound Marketing CRM — 리드 파이프라인 전체
> **목적**: 후속 이슈(P5-1 ~ P5-4)의 구현 기준이 되는 UX/IA 계약

---

## 0. 제품 목표

인플루언서·크리에이터 대상 아웃바운드 마케팅 CRM으로서,
**"탐색 → 검토 → 연락 대상 확정 → 이메일 발송"** 워크플로를 한 제품 안에서 끊김 없이 수행할 수 있게 한다.

핵심 가치:
- 사용자가 **현재 어떤 단계에 있는지** 항상 알 수 있어야 한다
- 각 화면에서 **무엇을 결정해야 하는지** 즉시 보여야 한다
- **이메일 보내기 직전에 볼 화면**이 단 하나여야 한다

## 1. 현재 문제 구조화

### 1.1 자연어 탐색 입력 파싱 불안정

| 항목 | 현재 상태 | 문제 |
|------|----------|------|
| 입력 | 자유 텍스트 프롬프트 | OK |
| 파싱 | `parsePrompt` API → 키워드/카테고리/구독자 범위 추출 | 파싱 실패 시 의미 없는 chip 나열 |
| 확인 | structured form 으로 편집 가능 | 파싱 결과와 원문 의도의 괴리가 크면 사용자가 수정 포기 |
| 실행 | 바로 Job 생성 | 파싱 검증 없이 실행되면 저품질 결과 |

**핵심 원인**: 파싱 결과를 사용자가 "이게 내 의도가 맞는지" 한눈에 판단할 수 있는 요약 surface가 없음.

### 1.2 리드 검토 화면의 목적 불명확

| 항목 | 현재 상태 | 문제 |
|------|----------|------|
| 탭 구조 | 8개 action_queue (needs_verification, contactable, needs_correction, held, excluded, conflict_queue, ready_to_sync, synced) | 운영 파이프라인 내부 상태를 그대로 탭으로 노출 → 사용자 목적과 불일치 |
| KPI 카드 | 13개 이상 메트릭 | 정보 과다. 사용자가 "지금 뭘 해야 하는지" 판단 불가 |
| Quality Banner | 자동 판단 + CTA | 배너와 탭의 관계 불명확 |

**핵심 원인**: "데이터 상태 보기 화면"이지 "결정 화면"이 아님.

### 1.3 상세 레이어 붕괴

| 레벨 | 현재 컴포넌트 | 문제 |
|------|-------------|------|
| L0 | ReviewTable (리스트) | OK |
| L1 | LeadDetailDrawer (우측 패널) | 빠른 확인용이지만 액션 버튼이 과다 |
| L2 | LeadFullDetail (중앙 모달) | drawer 위에 모달이 열림 → 맥락 끊김 |
| L3 | SupplementarySearchModal (모달) | 모달 위 모달 가능성 |

**핵심 원인**: side sheet → modal → modal 중첩으로 사용자가 현재 깊이를 잃음.

### 1.4 좌측 메뉴 정보구조 불일치

| 현재 메뉴 | 실제 역할 | 사용자 혼란 |
|-----------|----------|------------|
| 리드 파이프라인 (/collection) | 탐색 + 검토 + 운영자 도구 통합 | "파이프라인"이 무슨 뜻인지 불명확 |
| 전체 리드 (/leads) | raw lead DB 전체 보기 | 사용자가 여기서 뭘 해야 하는지 불명 |
| 마스터 리스트 (/master-list) | 중복 제거된 최종 리드 | "마스터"라는 명칭이 목적을 설명하지 못함 |

**핵심 원인**: 메뉴가 "데이터 저장소 위치"를 기준으로 구성되어 있고, "사용자 목적"을 기준으로 구성되지 않음.

## 2. 사용자 핵심 여정 (User Journey)

### 2.1 7단계 핵심 흐름

```
Step 1: 탐색 조건 입력
  └─ 자연어로 원하는 크리에이터 조건 입력

Step 2: 파싱 검토 & 확인
  └─ 시스템이 분해한 조건을 확인/수정 후 실행

Step 3: 탐색 실행 & 모니터링
  └─ 크롤링 진행 상황 확인, 필요 시 취소

Step 4: 결과 검토 & 분류
  └─ 수집된 리드를 빠르게 검토하여 분류
     (연락 가능 / 보강 필요 / 제외)

Step 5: 데이터 보강 (조건부)
  └─ 이메일 없음·프로필 미비 리드에 대해 보강 탐색

Step 6: 연락 대상 확정
  └─ 검토 완료 리드를 연락 대상 리스트로 이동

Step 7: 캠페인 투입
  └─ 연락 대상에서 캠페인 생성 → 이메일 발송
```

### 2.2 흐름도

```
[탐색 조건 입력] ──▶ [파싱 검토] ──▶ [탐색 실행]
                                         │
                                         ▼
                                   [결과 검토]
                                    │    │    │
                          ┌─────────┘    │    └──────────┐
                          ▼              ▼               ▼
                    [연락 가능]    [보강 필요]        [제외]
                          │              │
                          │              ▼
                          │        [보강 탐색]
                          │              │
                          ▼              ▼
                    [연락 대상 리스트] ◀──┘
                          │
                          ▼
                    [캠페인 투입]
                          │
                          ▼
                    [이메일 발송]
```

### 3.2 현재 → 개편 매핑

| 현재 메뉴 | 현재 경로 | 개편 후 | 개편 경로 | 변경 사유 |
|-----------|----------|---------|----------|----------|
| 리드 파이프라인 > 새 탐색 만들기 탭 | /collection (setup 탭) | **새 탐색** | /search/new | 독립 화면으로 승격. 탐색은 파이프라인 내 탭이 아니라 사용자의 시작점 |
| 리드 파이프라인 > 진행 중인 탐색 탭 | /collection (monitor 탭) | **진행 중인 탐색** | /search/active | 독립 화면으로 승격. 탐색 모니터링은 검토와 다른 목적 |
| 리드 파이프라인 > 리드 검토 탭 | /collection (results 탭) | **리드 검토** | /review | 독립 화면으로 승격. 제품의 핵심 결정 화면 |
| 전체 리드 | /leads | **운영자 도구 > 전체 리드 DB** | /admin/leads | 사용자 메인 워크플로에 불필요. 운영자 전용 |
| 마스터 리스트 | /master-list | **연락 대상** | /contacts | "마스터 리스트" → "연락 대상"으로 재명명. 목적 중심 명칭 |
| 리드 파이프라인 > 운영자 도구 탭 | /collection (operator 탭) | **운영자 도구 > 데이터 가져오기** | /admin/import | 운영 섹션으로 이동 |
| 설정 | /settings | **운영자 도구 > 시스템 설정** | /admin/settings | 일반 사용자가 자주 쓰지 않는 기능 |

### 3.3 화면별 목적 요약

| 화면 | 사용자 질문 | 핵심 행동 |
|------|-----------|----------|
| 대시보드 | "지금 상태가 어때?" | 현황 파악, 다음 행동 진입점 |
| 새 탐색 | "어떤 크리에이터를 찾을까?" | 조건 입력 → 파싱 확인 → 실행 |
| 진행 중인 탐색 | "탐색이 잘 되고 있나?" | 진행률 확인, 완료 알림, 취소 |
| 리드 검토 | "이 사람들 연락해도 되나?" | 빠른 분류: 연락 가능 / 보강 / 제외 |
| 연락 대상 | "이메일 보낼 사람 누구?" | 최종 리드 확인, 캠페인 투입 |
| 캠페인 | "이메일 보내자" | 캠페인 생성, 발송, 결과 확인 |
| 템플릿 | "어떤 메일을 보낼까?" | 템플릿 생성/편집 |
| 운영자 도구 | "데이터를 직접 확인/수정할 때" | raw data 조회, 가져오기, 설정 |

## 4. 용어 사전 (Glossary)

### 4.1 핵심 엔티티

| 용어 | 영문 | 정의 | 비고 |
|------|------|------|------|
| **리드** | Lead | 탐색을 통해 수집된 크리에이터/인플루언서 1건의 데이터. 아직 연락 대상으로 확정되지 않은 상태 | 시스템의 기본 데이터 단위 |
| **연락 대상** | Contact | 검토를 거쳐 연락 가능하다고 판단된 리드. 이메일 발송 대상 풀 | 구 "마스터 리스트" 항목 |
| **캠페인** | Campaign | 연락 대상 그룹에 이메일을 발송하는 단위 | 미구현 |
| **탐색** | Search Job | 특정 조건으로 크리에이터를 찾는 크롤링 작업 1회 | 구 "Job" |
| **보강 탐색** | Supplementary Search | 기존 탐색 결과의 부족한 데이터를 보완하는 추가 탐색 | 이메일 보강, 영향력 보강, 프로필 보강 |

### 4.2 "마스터 리스트" 처리 결정

| 항목 | 결정 |
|------|------|
| "마스터 리스트" 명칭 | **폐기**. "연락 대상(Contacts)"으로 변경 |
| 사유 | "마스터"는 데이터 관리자 관점 용어. 사용자 관점에서는 "이메일 보낼 사람 목록"이 목적이므로 "연락 대상"이 적절 |
| 기술적 변경 | `/master-list` → `/contacts`, `MasterList` 컴포넌트 → `Contacts` 컴포넌트, API 엔드포인트는 하위 호환 유지 |

### 4.3 "전체 리드" 처리 결정

| 항목 | 결정 |
|------|------|
| "전체 리드" 위치 | 사용자 메인 메뉴에서 **제거**. 운영자 도구 하위로 이동 |
| 사유 | raw lead 전체 DB는 일상 워크플로에 불필요. 사용자가 탐색 → 검토 → 연락 대상 흐름 안에서 필요한 리드만 접근하면 됨 |
| 접근 방법 | 운영자 도구 > 전체 리드 DB (/admin/leads) |

### 4.4 리드 단계별 용어

| 용어 | 영문 | 정의 | 현재 코드 매핑 |
|------|------|------|---------------|
| **수집된 리드** | Raw Lead | 탐색을 통해 수집만 된 상태. 아직 검토하지 않음 | review_status: pending \| auto_approved \| auto_rejected |
| **검토된 리드** | Reviewed Lead | 사용자가 확인하여 분류를 완료한 리드 | review_status: approved \| rejected \| held |
| **연락 가능 리드** | Campaign-Ready Lead | 검토 완료 + 이메일 유효 + 연락 대상으로 이동 가능한 상태 | master_sync_status: ready_to_sync \| synced |

### 4.5 운영 상태 용어

| UI 표시 용어 | 영문 | 정의 | 사용자 행동 |
|-------------|------|------|------------|
| **검토 필요** | Needs Review | 시스템이 자동 판단하지 못하여 사용자 확인이 필요한 리드 | 상세 확인 후 분류 |
| **보강 필요** | Needs Enrichment | 이메일 없음, 프로필 미비 등 데이터가 부족한 리드 | 보강 탐색 실행 또는 수동 입력 |
| **연락 가능** | Contactable | 이메일 유효, 프로필 충분. 연락 대상으로 이동 가능 | "연락 대상으로 이동" 실행 |
| **제외** | Excluded | 부적합하여 더 이상 진행하지 않는 리드 | 없음 (최종 상태) |
| **보류** | On Hold | 당장은 아니지만 나중에 다시 검토할 리드 | 나중에 재검토 |
| **연락 대상 이동 완료** | Synced to Contacts | 연락 대상 리스트에 반영 완료 | 캠페인에서 사용 가능 |

### 4.6 데이터 상태 용어

| UI 표시 용어 | 영문 | 정의 | 코드 매핑 |
|-------------|------|------|----------|
| **이메일 없음** | No Email | 이메일 주소가 수집되지 않음 | email_status: missing |
| **이메일 미검증** | Email Unverified | 이메일은 있으나 유효성 미확인 | email_status: unverified |
| **이메일 유효** | Email Valid | 이메일 형식 검증 통과 | email_status: valid_syntax |
| **이메일 무효** | Email Invalid | 이메일 형식 불량 | email_status: invalid_syntax |
| **이메일 수정됨** | Email Corrected | 사용자가 직접 수정한 이메일 | email_status: user_corrected |
| **플랫폼 메일 의심** | Platform Email Suspect | 개인 이메일이 아닌 플랫폼 생성 이메일로 추정 | contact_readiness: platform_suspect |
| **프로필 미보강** | Not Enriched | 보강 데이터 수집 안 됨 | enrichment_status: not_started |
| **보강 완료** | Enriched | 보강 데이터 수집 완료 | enrichment_status: completed |
| **보강 신뢰도 낮음** | Low Confidence Enrichment | 보강 데이터가 있으나 신뢰도 낮음 | enrichment_status: low_confidence |
| **중복 의심** | Duplicate Suspect | 연락 대상 리스트에 유사 항목 존재 | conflict_details 존재 |

## 5. 상태 모델 (State Model)

### 5.1 두 축 분리 원칙

리드의 상태를 **데이터 품질 상태**와 **운영 파이프라인 상태**, 두 축으로 분리한다.
이 두 축은 독립적으로 관리되며, 화면에서는 둘 다 표시하되 **운영 상태가 primary**, 데이터 상태가 secondary이다.

### 5.2 데이터 품질 상태 (Data Quality State)

데이터 자체의 완결성을 나타낸다. 시스템이 자동으로 평가한다.

```
┌─────────────────────────────────────────────┐
│  이메일 상태    │  없음 → 미검증 → 유효/무효  │
│  (email_status) │         → 수정됨            │
├─────────────────┼────────────────────────────┤
│  연락 가능성      │  이메일 없음               │
│  (contact_       │  플랫폼 메일 의심           │
│   readiness)     │  검증 필요                 │
│                  │  연락 가능                  │
│                  │  사용자 확인                │
├─────────────────┼────────────────────────────┤
│  보강 상태       │  미시작 → 완료             │
│  (enrichment_   │       → 신뢰도 낮음         │
│   status)        │       → 실패               │
├─────────────────┼────────────────────────────┤
│  영향력 등급     │  Nano/Micro/Mid/Macro/Mega │
│  (audience_tier) │                            │
├─────────────────┼────────────────────────────┤
│  중복 여부       │  conflict_details 존재 여부 │
│  (conflict)      │                            │
└─────────────────┴────────────────────────────┘
```

### 5.3 운영 파이프라인 상태 (Operational Pipeline State)

사용자가 리드를 어디까지 처리했는지를 나타낸다.

```
[수집됨] ──▶ [검토 필요] ──▶ [연락 가능] ──▶ [연락 대상 이동 대기] ──▶ [연락 대상 반영 완료]
                │                                    ▲
                ├──▶ [보강 필요] ─── (보강 후) ──────┘
                ├──▶ [보류]
                └──▶ [제외]

세부 상태 전이:

1. 수집됨 (pending)
   → 시스템 자동 분류
   → 검토 필요 (needs_review) 또는 자동 승인/제외

2. 검토 필요 (needs_review)
   → 사용자 판단
   → 연락 가능 (approved) | 보강 필요 (needs_correction) | 보류 (held) | 제외 (rejected)

3. 연락 가능 (approved)
   → 중복 검사
   → 충돌 없음: 이동 대기 (ready_to_sync)
   → 충돌 있음: 충돌 확인 (conflict_queue) → 해결 후 이동 대기 | 제외

4. 이동 대기 (ready_to_sync)
   → 사용자 최종 확인
   → 연락 대상 반영 완료 (synced)

5. 보강 필요 (needs_correction)
   → 보강 탐색 실행
   → 결과 반영 후 → 검토 필요로 복귀

6. 보류 (held)
   → 나중에 재검토 → 검토 필요로 복귀 가능

7. 제외 (rejected)
   → 최종 상태. 복구 가능하나 의도적 조작 필요
```

### 5.4 현재 탭 → 개편 탭 매핑

현재 ReviewWorkspace의 8개 action_queue 탭을 사용자 목적 기반 3+1 탭으로 축소한다.

| 개편 탭 | 포함하는 현재 큐 | 목적 |
|---------|----------------|------|
| **검토 필요** | needs_verification + needs_correction | 사용자가 판단을 내려야 하는 리드 |
| **연락 가능** | contactable + ready_to_sync + conflict_queue | 연락 대상 후보. 충돌 있는 것은 인라인 표시 |
| **제외/보류** | excluded + held | 더 이상 진행하지 않는 리드 |
| **반영 완료** (접힘) | synced | 아카이브. 평소에는 숨김 |

## 6. Interaction Grammar

### 6.1 레이어 규칙

| 레벨 | Surface | 용도 | 허용 |
|------|---------|------|------|
| L0 | **리스트 (테이블)** | 데이터 스캔, 선택, bulk action | 항상 |
| L1 | **Side Sheet (우측 패널)** | quick review. 핵심 정보 + 1-2개 primary action | row 클릭 시 |
| L2 | **Full Page Detail** | 깊은 편집, 이력 확인, 출처 검증 | side sheet 내 "상세 보기" 링크 클릭 시. 별도 URL 부여 |
| L3 | **Dialog (모달)** | 되돌릴 수 없는 확인만 | 제외 확인, 일괄 삭제 확인, 충돌 해결 확인 |

**금지 규칙:**
- Side sheet 위에 모달을 띄우지 않는다 (현재 LeadFullDetail이 이를 위반)
- 모달 위에 모달을 띄우지 않는다
- 리스트 → side sheet → full page 이상의 깊이를 허용하지 않는다

### 6.2 Row Click 행동

| 화면 | Row Click | 결과 |
|------|-----------|------|
| 리드 검토 | 단일 클릭 | Side sheet 열림 (해당 리드 quick review) |
| 리드 검토 | 체크박스 | 선택 상태 토글 (bulk action용) |
| 연락 대상 | 단일 클릭 | Side sheet 열림 (연락 정보 중심) |
| 진행 중인 탐색 | 단일 클릭 | 해당 탐색 상세 패널 확장 (accordion) |
| 전체 리드 DB (운영자) | 단일 클릭 | Side sheet 열림 (raw data 중심) |

### 6.3 Side Sheet 규칙

```
┌──────────────────────────────────────────────┐
│  Side Sheet 구조                              │
│                                               │
│  ┌─ 헤더 ─────────────────────────────────┐  │
│  │  채널명 / 이름                          │  │
│  │  플랫폼 뱃지  ·  영향력 등급            │  │
│  │  [상세 보기 →]  (full page 이동 링크)   │  │
│  └────────────────────────────────────────┘  │
│                                               │
│  ┌─ 핵심 정보 ────────────────────────────┐  │
│  │  이메일: xxx@xxx.com (상태 뱃지)       │  │
│  │  구독자: 12.5K                          │  │
│  │  채널 URL                               │  │
│  └────────────────────────────────────────┘  │
│                                               │
│  ┌─ 현재 상태 ────────────────────────────┐  │
│  │  데이터 상태: [이메일 유효] [보강 완료]  │  │
│  │  운영 상태: 검토 필요                    │  │
│  │  (이유가 있으면 간단히 표시)             │  │
│  └────────────────────────────────────────┘  │
│                                               │
│  ┌─ Primary Action ───────────────────────┐  │
│  │  [연락 대상으로 이동]  ← primary CTA    │  │
│  │  [보강 필요]  [보류]  [제외]             │  │
│  └────────────────────────────────────────┘  │
│                                               │
└──────────────────────────────────────────────┘
```

**Side sheet에서 보여주지 않을 것:**
- 보강 상세 데이터 (enrichment 전체 JSON)
- 편집 이력 (history)
- 출처/근거 상세 (source details)
- 인라인 편집 (편집은 full page에서)

### 6.4 Full Page Detail 규칙

- 별도 URL을 가진다: `/review/:leadId` 또는 `/contacts/:leadId`
- side sheet에서 "상세 보기" 클릭 시 이동한다
- 현재 LeadFullDetail의 4개 탭 구조를 유지하되, 모달이 아닌 full page로 변경한다
- 브라우저 뒤로 가기로 리스트에 복귀 가능해야 한다
- 편집은 여기서만 허용한다

### 6.5 Dialog(모달) 허용 조건

| 허용 | 예시 |
|------|------|
| 되돌리기 어려운 액션 확인 | "선택한 15건을 제외하시겠습니까?" |
| 파괴적 액션 확인 | "이 리드를 연락 대상에서 삭제하시겠습니까?" |
| 충돌 해결 선택 | "기존 데이터를 유지하시겠습니까, 새 데이터로 교체하시겠습니까?" |

| 금지 | 현재 위반 |
|------|----------|
| 상세 정보 표시 | LeadFullDetail이 모달로 열림 |
| 보강 탐색 설정 | SupplementarySearchModal |
| 정보 입력 폼 | 없음 (현재는 해당 없음) |

### 6.6 Bulk Action 규칙

| 위치 | 행동 | 트리거 |
|------|------|--------|
| 리드 검토 > 검토 필요 탭 | 일괄 승인, 일괄 보류, 일괄 제외 | 체크박스 선택 후 상단 액션 바 |
| 리드 검토 > 연락 가능 탭 | 일괄 연락 대상 이동 | 체크박스 선택 후 상단 액션 바 |
| 연락 대상 | 일괄 캠페인 투입, 일괄 삭제 | 체크박스 선택 후 상단 액션 바 |

**Bulk action은 항상 확인 dialog를 거친다.**

### 6.7 화면별 Primary CTA

| 화면 | Primary CTA | 위치 |
|------|------------|------|
| 대시보드 | 새 탐색 시작 | 상단 퀵 액션 |
| 새 탐색 | 탐색 실행 | 폼 하단 |
| 진행 중인 탐색 | 결과 보기 (완료된 탐색) | 각 탐색 카드 내 |
| 리드 검토 > 검토 필요 | 연락 가능으로 이동 | side sheet 및 bulk action 바 |
| 리드 검토 > 연락 가능 | 연락 대상으로 이동 | side sheet 및 bulk action 바 |
| 연락 대상 | 캠페인 만들기 | 상단 또는 bulk action 바 |
| 캠페인 | 발송 시작 | 캠페인 상세 내 |

## 7. Handoff 구조

### 7.1 리드 handoff 정의

"handoff"란 리드가 한 화면의 관할에서 다른 화면의 관할로 이동하는 것을 의미한다.

| From | Action | To | 조건 |
|------|--------|----|------|
| 탐색 결과 | 탐색 완료 | 리드 검토 | 자동. 탐색 완료 시 리드 검토에 표시 |
| 리드 검토 | "연락 대상으로 이동" | 연락 대상 | 수동. 사용자 승인 + 중복 검사 통과 |
| 연락 대상 | "캠페인 투입" | 캠페인 | 수동. 사용자가 캠페인에 추가 |
| 리드 검토 | "제외" | (사라짐) | 수동. 제외 탭에서만 확인 가능 |

### 7.2 Handoff 시 중복 검사

리드가 "연락 대상"으로 이동할 때 반드시 중복 검사를 수행한다.

```
[연락 대상으로 이동] 클릭
  │
  ▼
중복 검사 (이메일 + 채널 URL 기준)
  │
  ├── 충돌 없음 → 바로 이동
  │
  └── 충돌 발견 → 충돌 배너 표시
       │
       ├── "기존 유지" → 새 리드 제외
       └── "새 데이터로 교체" → 기존 항목 업데이트
```

### 7.3 되돌리기 규칙

| 동작 | 되돌리기 |
|------|---------|
| 검토 → 연락 가능 | 가능. 리드 검토에서 상태 변경 |
| 연락 가능 → 연락 대상 이동 | 가능. 연락 대상에서 "리드 검토로 되돌리기" |
| 제외 | 가능. 제외 탭에서 "재검토" |
| 연락 대상 → 캠페인 투입 | 캠페인 미발송 시 가능, 발송 후 불가 |

## 8. 화면별 역할 정의 (Screen Contracts)

### 8.1 대시보드 (`/`)

| 항목 | 정의 |
|------|------|
| **목적** | 전체 현황 파악 + 다음 행동 진입점 |
| **사용자가 내려야 하는 결정** | "지금 무엇을 해야 하는가?" — 새 탐색 / 검토 / 연락 대상 확인 |
| **Primary CTA** | 새 탐색 시작 |
| **최소 정보** | 검토 대기 리드 수, 연락 가능 리드 수, 진행 중인 탐색 수, 최근 활동 |
| **보여주지 않을 정보** | raw lead 통계, 기술적 크롤링 상세, enrichment 메트릭 |

### 8.2 새 탐색 (`/search/new`)

| 항목 | 정의 |
|------|------|
| **목적** | 찾고 싶은 크리에이터 조건을 정의하고, 파싱 결과를 확인한 후 탐색 실행 |
| **사용자가 내려야 하는 결정** | "이 파싱 결과가 내 의도와 맞는가?" → 수정 또는 실행 |
| **Primary CTA** | 탐색 실행 (파싱 확인 완료 후에만 활성화) |
| **최소 정보** | 자연어 입력란, 파싱 결과 요약 카드 (키워드, 카테고리, 구독자 범위, 플랫폼), 예시 프롬프트 |
| **보여주지 않을 정보** | 이전 탐색 결과, 기술적 크롤링 설정 (max retries, delay 등은 고급 설정으로 접힘) |

### 8.3 진행 중인 탐색 (`/search/active`)

| 항목 | 정의 |
|------|------|
| **목적** | 실행 중인 탐색의 진행 상태 확인, 완료된 탐색의 결과 이동 |
| **사용자가 내려야 하는 결정** | "이 탐색을 계속 둘까, 취소할까?" / "결과를 검토하러 갈까?" |
| **Primary CTA** | 결과 보기 (완료된 탐색에 표시) |
| **최소 정보** | 탐색 조건 요약, 진행률, 수집된 리드 수, 상태 (실행 중/완료/실패) |
| **보여주지 않을 정보** | 개별 리드 데이터, enrichment 상세 |

### 8.4 리드 검토 (`/review`)

| 항목 | 정의 |
|------|------|
| **목적** | 수집된 리드를 빠르게 검토하여 분류하는 **결정 화면** |
| **사용자가 내려야 하는 결정** | 각 리드에 대해 "연락 가능 / 보강 필요 / 제외 / 보류" |
| **Primary CTA** | 연락 대상으로 이동 (연락 가능 탭) / 분류하기 (검토 필요 탭) |
| **최소 정보** | 채널명, 플랫폼, 이메일 유무/상태, 구독자 수, 영향력 등급, 현재 문제 (있을 경우) |
| **보여주지 않을 정보** | enrichment JSON, 크롤링 소스 상세, 편집 이력 (side sheet에서 제외, full page에서 제공) |

**탭 구조:**

| 탭 | 뱃지 | 내용 |
|----|------|------|
| 검토 필요 | 건수 | 사용자가 판단해야 하는 리드 |
| 연락 가능 | 건수 | 승인된 리드. "연락 대상으로 이동" 가능 |
| 제외/보류 | — | 이전에 제외/보류한 리드. 필요 시 재검토 |

**좌측 사이드바 (탐색 선택):**
- 완료된 탐색 목록
- 탐색을 선택하면 해당 탐색의 리드만 표시
- "전체" 선택 시 모든 탐색의 미처리 리드 표시

### 8.5 연락 대상 (`/contacts`)

| 항목 | 정의 |
|------|------|
| **목적** | **이메일 보내기 직전에 보는 단일 화면**. 연락 가능으로 확정된 리드의 최종 리스트 |
| **사용자가 내려야 하는 결정** | "이 사람들에게 캠페인을 보낼까?" |
| **Primary CTA** | 캠페인 만들기 (선택된 연락 대상 기반) |
| **최소 정보** | 이름/채널명, 이메일, 플랫폼, 영향력 등급, 마지막 연락 상태 (있을 경우) |
| **보여주지 않을 정보** | 검토 이력, 수집 소스, raw data. 이미 검증 완료된 데이터만 표시 |

### 8.6 캠페인 (`/campaigns`)

| 항목 | 정의 |
|------|------|
| **목적** | 이메일 캠페인 생성, 발송, 결과 추적 |
| **사용자가 내려야 하는 결정** | "이 캠페인을 발송할까?" / "결과가 어떤가?" |
| **Primary CTA** | 새 캠페인 만들기 / 발송 시작 |
| **최소 정보** | 캠페인명, 대상 수, 발송 상태, 오픈율, 답변율 |
| **보여주지 않을 정보** | 개별 리드의 수집/검토 과정 |

### 8.7 운영자 도구 > 전체 리드 DB (`/admin/leads`)

| 항목 | 정의 |
|------|------|
| **목적** | raw lead 전체를 기술적으로 조회/관리하는 운영자 전용 화면 |
| **사용자가 내려야 하는 결정** | 데이터 이상 확인, 수동 보정, 디버깅 |
| **Primary CTA** | CSV 내보내기 |
| **최소 정보** | 모든 필드 (raw data 전체) |
| **보여주지 않을 정보** | 없음. 운영자에게는 모든 데이터 노출 |

### 8.8 운영자 도구 > 데이터 가져오기 (`/admin/import`)

| 항목 | 정의 |
|------|------|
| **목적** | 외부 데이터 (enrichment, 키워드) 를 시스템에 가져오기 |

## 9. 후속 이슈별 입력 조건

### 9.1 P5-1: 자연어 탐색 입력 / 파서

**범위**: `/search/new` 화면

**이 문서에서 확정된 규칙:**
- 자연어 입력 후 **즉시 실행 금지**
- 파싱 결과를 **요약 카드** 형태로 먼저 보여준다
- 요약 카드에 포함: 키워드, 카테고리, 구독자 범위, 플랫폼 힌트
- 사용자가 확인/수정한 후에만 "탐색 실행" 버튼 활성화
- chip은 "입력 자체"가 아니라 "파싱 결과 편집"에만 사용

**현재 코드 참조:**
- `CollectionSetupForm.tsx` — 현재 3단계 플로우 (Input → Parsing → Confirm) 존재하나, 파싱 실패 시 UX 부족
- `api/jobs.ts > parsePrompt()` — 파싱 API

**구현 방향:**
1. 파싱 실패/부분 실패 시 사용자에게 "이 부분을 이해하지 못했습니다" 피드백 제공
2. 파싱 결과 요약 카드를 structured form 전에 중간 단계로 삽입
3. 고급 설정 (max retries, delay 등)은 접힘 패널로 숨김

### 9.2 P5-2: 리드 검토 인박스

**범위**: `/review` 화면

**이 문서에서 확정된 규칙:**
- 탭을 8개 → 3+1개로 축소 (검토 필요 / 연락 가능 / 제외·보류 / 반영 완료)
- KPI 카드를 최소화: 검토 대기 건수, 연락 가능 건수, 보강 필요 건수 정도
- Quality Banner는 보강 필요 탭에서만 표시
- side sheet는 quick review용. 편집 불가
- bulk action은 상단 액션 바에서 수행

**현재 코드 참조:**
- `ReviewWorkspace.tsx` — 메인 검토 화면
- `ReviewTable.tsx` — 리드 테이블
- `ReviewKPICards.tsx` — KPI 카드 (축소 필요)
- `QualityBanner.tsx` — 품질 배너
- `BulkActions.tsx` — 일괄 액션
- `LeadDetailDrawer.tsx` — side sheet (경량화 필요)
- `ReviewJobSidebar.tsx` — 탐색 선택 사이드바

**구현 방향:**
1. action_queue 8개를 사용자 관점 3+1 탭으로 묶는 매핑 레이어 추가
2. KPI 카드를 3개 핵심 숫자로 축소
3. side sheet에서 편집 기능 제거, "상세 보기" 링크 추가
4. LeadFullDetail을 모달에서 full page로 전환

### 9.3 P5-3: 리드 상세 Surface

**범위**: `/review/:leadId`, `/contacts/:leadId` 페이지

**이 문서에서 확정된 규칙:**
- side sheet 위에 모달 띄우기 금지
- 상세 보기는 full page (별도 URL)
- 편집은 full page에서만
- 4탭 구조 유지 (프로필, 연락처/영향력, 출처·근거, 보정이력)

**현재 코드 참조:**
- `LeadFullDetail.tsx` — 현재 모달. full page로 전환 필요
- `LeadDetailDrawer.tsx` — side sheet. 경량화 필요
- `EditableField.tsx` — 편집 컴포넌트. full page에서만 사용하도록 제한

**구현 방향:**

2. 라우트 추가: `/review/:leadId`, `/contacts/:leadId`

### 9.4 P5-4: 메뉴 / 리스트 / Handoff 구조

**범위**: Sidebar, 라우팅, 화면 간 이동
